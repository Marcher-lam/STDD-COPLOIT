# STDD Certainty Protocol (/stdd:certainty)

目标：借鉴 Claude Pilot 的 Certainty Protocol，在 STDD 工作流的关键决策点进行置信度评分。当 AI 的置信度低于阈值时，自动暂停并请求人类确认，防止 AI 在不确定时盲目推进。

## 核心原则

> **不确定就停下来** — 宁可多问一次，也不要在错误的方向上狂奔。

---

## 置信度评分模型

### 评分维度（每个 0-1 分）

| 维度 | 说明 | 评估方式 |
|------|------|---------|
| **需求清晰度** | 需求是否足够明确无歧义 | 是否存在模糊词汇、未定义边界 |
| **技术可行性** | 选定方案在当前技术栈下是否可行 | 是否依赖不存在的 API、已废弃的库 |
| **上下文完整度** | 是否拥有足够的代码/架构上下文 | 是否遗漏了关键文件、未读取 contracts |
| **测试可靠性** | 测试是否真正验证了期望行为 | 是否存在骗绿灯断言、Mock 过度 |
| **一致性** | 实现是否与 BDD 规格一致 | 是否实现了规格外的功能、遗漏了规格内的功能 |

### 综合置信度

```
certainty = min(需求清晰度, 技术可行性, 上下文完整度, 测试可靠性, 一致性)
```

> 取最小值而非平均值：任何单一维度低置信度都应触发审查。

---

## 阈值体系

| 阈值 | 值 | 行为 |
|------|-----|------|
| **confirm_threshold** | 0.7 | 低于此值 → 暂停并请求人类确认 |
| **warning_threshold** | 0.85 | 低于此值 → 输出警告但继续执行 |
| **auto_threshold** | 0.95 | 高于此值 → 自动推进不中断 |

## 触发点（Check Points）

在 STDD 工作流的以下关键节点自动触发 Certainty 评估：

### CP-1: Spec 生成后（stdd-spec）

```
评估:
  需求清晰度: 需求是否已充分澄清？
  一致性:     BDD 规格是否完整覆盖 proposal？

if certainty < 0.7:
  暂停: "BDD 规格可能存在遗漏，请审阅以下不确定点:"
  列出低置信度维度及原因
```

### CP-2: Plan 拆解后（stdd-plan）

```
评估:
  技术可行性: 任务是否在当前技术栈下可实现？
  上下文完整度: 是否已读取所有相关文件？
  一致性:     任务拆解是否覆盖所有 BDD 场景？

if certainty < 0.7:
  暂停: "任务拆解可能不够完整或存在技术风险:"
  列出不确定的技术点和遗漏的上下文
```

### CP-3: Red 阶段（stdd-execute 写测试时）

```
评估:
  需求清晰度: 当前微任务的期望行为是否明确？
  测试可靠性: 测试是否精确验证了期望行为？

if certainty < 0.7:
  暂停: "测试可能不够精确:"
  提示用户确认测试断言是否正确
```

### CP-4: Green 阶段（实现代码后）

```
评估:
  一致性:     实现是否恰好满足测试要求？
  技术可行性: 实现方式是否合理？

if certainty < 0.7:
  暂停: "实现可能超出或偏离测试要求:"
  列出可能的超纲代码
```

### CP-5: Refactor 阶段（重构后）

```
评估:
  测试可靠性: 重构后测试是否仍然通过？
  一致性:     重构是否改变了可观察行为？

if certainty < 0.7:
  暂停: "重构可能改变了行为:"
  提示用户确认重构范围
```

## 与 Ralph Loop 集成

Certainty Protocol 嵌入 Ralph Loop 的每个阶段：

```
┌──────────────────────────────────────────────────────────────────┐
│                    Ralph Loop + Certainty                        │
│                                                                  │
│  1. 🔴 RED                                                      │
│     └──► CP-3 评估 → certainty < 0.7 ? 暂停 : 继续             │
│                                                                  │
│  2. 🔍 CHECK (静态质检)                                          │
│     └──► 自动通过（质检是确定性的）                               │
│                                                                  │
│  3. 🟢 GREEN                                                    │
│     └──► CP-4 评估 → certainty < 0.7 ? 暂停 : 继续             │
│                                                                  │
│  4. 🧪 MUTATION (伪变异审查)                                    │
│     └──► 测试可靠性维度自动计算                                  │
│                                                                  │
│  5. 🔵 REFACTOR                                                 │
│     └──► CP-5 评估 → certainty < 0.7 ? 暂停 : 继续             │
│                                                                  │
│  ⚠️ 连续失败 3 次 → 熔断                                       │
│  ⚠️ certainty 连续 < 0.7 → 同样触发熔断                        │
└──────────────────────────────────────────────────────────────────┘
```

### 与 HITL 的协同

| HITL 模式 | Certainty 行为 |
|-----------|---------------|
| `every-phase` | 每个阶段都评估，低置信度暂停 |
| `end-of-cycle` | 每个循环结束时评估一次 |
| `off` | 仅在 certainty < 0.5（远低于阈值）时暂停 |

---

## 配置

在 `stdd/config.yaml` 中：

```yaml
tdd:
  certainty:
    enabled: true
    # 阈值
    confirm_threshold: 0.7    # 低于此值暂停
    warning_threshold: 0.85   # 低于此值警告
    auto_threshold: 0.95      # 高于此值自动推进
    # 启用的检查点
    check_points:
      - cp1_spec       # BDD 规格生成后
      - cp2_plan       # 任务拆解后
      - cp3_red        # 写测试时
      - cp4_green      # 实现后
      - cp5_refactor   # 重构后
    # 连续低置信度处理
    consecutive_low_max: 3    # 连续 3 次低置信度 → 熔断
```

## 快速参考

```bash
# 手动评估当前状态
/stdd:certainty

# 评估特定检查点
/stdd:certainty cp3_red

# 查看历史置信度趋势
/stdd:certainty trend --period=7d

# 跳过当前检查点（用户明确确认）
/stdd:certainty skip --reason="已手动验证"
```
