# STDD 需求确认向导 (/stdd:confirm)

目标：作为人机确认门 (Human Confirm Gate)，让用户最终审阅并批准需求，防止 AI 理解偏差带入后续阶段。

## 前置条件

- `proposal.md` 存在且包含澄清记录

## 执行步骤

### 1. 读取提案

加载 `proposal.md`，提取：
- 原始需求正文
- 澄清记录（`<!-- Clarify -->` 区块）
- 范围（包含/不包含）

### 2. 生成确认报告

自动生成结构化的需求确认报告，输出到终端：

```
╔══════════════════════════════════════════════════════╗
║              📋 Human Confirm Gate                    ║
╠══════════════════════════════════════════════════════╣
║                                                       ║
║  原始需求:                                            ║
║    实现一个支持 Markdown 导出的 todo-list              ║
║                                                       ║
║  澄清记录:                                            ║
║    Round #1:                                          ║
║    - 持久化: IndexedDB                                ║
║    - 空列表提示: 禁用按钮 + 提示文案                   ║
║                                                       ║
║  范围:                                                ║
║    ✅ 包含: 添加/删除/完成/导出/持久化                  ║
║    ❌ 不含: 多用户/云同步/拖拽排序                      ║
║                                                       ║
║  确认？(yes/no)                                       ║
╚══════════════════════════════════════════════════════╝
```

### 3. 等待用户确认

暂停执行，等待用户输入 **`yes`** 或 **`no`**。

### 4. 分支处理

| 用户输入 | 动作 |
|----------|------|
| **yes** | 在 `proposal.md` 追加 `<!-- Confirmed at <timestamp> -->`，输出确认通过，**自动调用** `/stdd:spec` |
| **no** | 在 `proposal.md` 追加 `<!-- Rejected at <timestamp> -->`，输出拒绝原因提示，**自动调用** `/stdd:propose` |
| **partial** | 记录用户修改意见，更新 proposal.md，重新进入步骤 2 |

## 确认门控原则

这是 **5 级防跑偏** 的第 1 级 —— 关键决策需人类确认。此门一旦通过，后续 Spec/Plan/Execute 阶段将严格基于已确认的需求，不再回头修改。

## 输出

- 更新后的 `proposal.md`（含 Confirmed 或 Rejected 标记）

## 边界情况

| 情况 | 处理方式 |
|------|----------|
| 无澄清记录 | 警告用户"未经澄清"，但仍允许确认 |
| 多次确认 | 只保留最后一次确认标记 |
| 用户超时无响应 | 保持挂起状态，可随时恢复 |

## 与其他 Skill 的关系

```
/stdd:clarify ──► /stdd:confirm ──► (yes) /stdd:spec
                              ──► (no)  /stdd:propose
```
