# STDD TDD 执行循环引擎 (/stdd:execute)

目标：严格死守 "红灯 → 静态检查 → 绿灯 → 伪变异审查 → 重构" 的隔离微执行护城河。

## 前置条件

- `tasks.md` 已生成且包含 Checkbox 任务列表
- `contracts.md`（或共享类型文件）已存在

## Ralph Loop 执行流程

对 `tasks.md` 中的每个未完成任务 `[ ]`，依次执行以下闭环：

**HITL 粒度配置**（来自 `stdd/config.yaml` → `tdd.hitl.mode`）：

| 模式 | 暂停时机 | 适用场景 |
|------|---------|---------|
| `every-phase` | 每个 Red/Green/Refactor 阶段后 | 高风险功能、安全敏感 |
| `end-of-cycle` | 每个完整循环后（默认） | 日常开发 |
| `off` | 仅熔断时暂停 | turbo 模式、低风险任务 |

```
┌──────────────────────────────────────────────────────────┐
│                    Ralph Loop (单任务)                      │
│                                                            │
│  1. 🔴 RED      → 写必然失败的测试（禁止写实现代码）        │
│  2. 🔍 CHECK    → 静态语法/类型检查（tsc/eslint/mypy）     │
│  3. 🟢 GREEN    → 最简实现让测试通过（禁止超纲）            │
│  4. 🧪 MUTATE   → 伪变异审查（检测骗绿灯断言）            │
│  5. 🔵 REFACTOR → 优化代码结构                              │
│  6. ✅ DONE     → 打勾 [x]，进入下一个任务                  │
│                                                            │
│  ⚠️ 连续失败 3 次 → 熔断（提示用户人工判断）               │
└──────────────────────────────────────────────────────────┘
```

### 阶段详解

#### 1. Red 阶段（聚光灯隔离）

- 从 `tasks.md` 取出 **1 个**微任务
- **必须读取** `contracts.md` 校准接口类型
- 生成必然失败的测试文件
- **绝对红线**：不可顺带写被测功能的任何实现代码
- 运行测试，确认红灯（测试全部失败 = 正确）

#### 2. 静态质检（Lint/Type Gate）

运行当前技术栈的静态检查命令：

| 语言 | 命令 |
|------|------|
| TypeScript | `tsc --noEmit` |
| Python | `mypy .` |
| Java | `mvn compile` |
| Go | `go vet ./...` |

- 若有低级报错 → 在此阶段直接修复（不算失败次数）
- 修复后重新运行检查

#### 3. Green 阶段（婴儿步）

- 极度吝啬地只写让测试 **苟且通过** 的最简代码
- **严厉禁止** 自行补充超纲要求（"顺便加个验证"、"顺手优化下" 等）
- 运行测试确认绿灯

#### 4. 伪变异审查（AI Assertion Review）

测试亮绿灯后，强制自检刚才写的测试：

**检测"骗绿灯断言"**：
- `expect(res).toBeTruthy()` — 太宽泛，打回重写
- `expect(res).toBeDefined()` — 没验证具体值，打回
- 全是 Mock 返回固定值，没有真正测试逻辑，打回
- `expect(error).toBeNull()` 但没有构造会触发 error 的场景，打回

**修正动作**：发现骗绿灯断言 → 删除该断言 → 写入有意义的断言 → 重跑测试

#### 5. Refactor 阶段

安全审查通过后：
- 优化模块结构（提取公共方法、消除重复）
- 运行全量测试确认未破坏
- 更新 `tasks.md` 勾选 `[x]`

## 熔断机制

```
失败计数器:
  连续失败 0 次 → 正常继续
  连续失败 1 次 → 自动重试（调整策略）
  连续失败 2 次 → 跨模型降级委托 (Cross-Model Delegation)
  连续失败 3 次 → 🔴 熔断!

跨模型降级委托（参考 Claude Pilot）:
  当连续失败 2 次时，自动切换到不同 AI 模型/策略尝试解决:

  ┌─────────────────────────────────────────────────────────┐
  │             Cross-Model Delegation Strategy                │
  ├─────────────────────────────────────────────────────────┤
  │                                                          │
  │  失败 1 次: 策略调整                                    │
  │  └── 简化方案、缩小测试范围、降低复杂度                  │
  │                                                          │
  │  失败 2 次: 模型降级                                    │
  │  └── 切换推理策略:                                       │
  │      ├── Claude Opus → Claude Sonnet (速度换质量)        │
  │      ├── 完整 TDD → Baby Step TDD (更小步)               │
  │      ├── 复杂断言 → 简单断言 (降低断言复杂度)            │
  │      └── 整体实现 → 分步实现 (拆分为更小单元)            │
  │                                                          │
  │  失败 3 次: 🔴 熔断                                     │
  │  └── 交还用户控制权                                      │
  │                                                          │
  └─────────────────────────────────────────────────────────┘

熔断动作:
  1. 停止当前任务的 Ralph Loop
  2. 输出错误详情和已尝试的方案
  3. 输出降级委托尝试记录（哪些策略/模型被尝试过）
  4. 提示用户人工判断:
     - 手动 git stash 回退
     - 修改任务拆解
     - 提供更多上下文
  5. 等待用户指示后恢复
```

## 输出

- `tasks.md`（所有任务勾选 `[x]`）
- `src/` 下的实现代码
- `__tests__/` 下的测试代码
- `implementation_log.md`（执行日志）

## 边界情况

| 情况 | 处理方式 |
|------|----------|
| 无 contracts.md | 警告但继续执行，从 tasks.md 推断接口 |
| 测试框架未配置 | 使用项目已有的测试框架（检测 package.json） |
| 任务间有依赖 | 按 tasks.md 顺序执行，跳过被依赖任务 |
| 单个任务过大 | 拆分为子任务后继续 |

## 双测试流约束

参考 ATDD，Ralph Loop 中的 CHECK 阶段强制执行双测试流：

```
┌─────────────────────────────────────────────────────┐
│               Dual Test Stream                       │
├─────────────────────────────────────────────────────┤
│                                                      │
│   Stream 1: 功能测试 (Functional)                    │
│   ├── 验证业务逻辑正确性                              │
│   ├── BDD Scenario → 测试映射                        │
│   └── 每个微任务必须有对应功能测试                     │
│                                                      │
│   Stream 2: 质量测试 (Quality)                       │
│   ├── 变异测试 (stdd-mutation)                       │
│   ├── 复杂度检查 (stdd-complexity)                    │
│   ├── 规范合规 (stdd-validate)                       │
│   └── 安全审查 (stdd-validate --review --focus=security) │
│                                                      │
│   双流必须全部通过才能进入下一循环                      │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 双流检查点

```
Ralph Loop: Red → Green → [CHECK] → Refactor
                         │
                         ├──► Stream 1: 功能测试通过？
                         │    └── 否 → 回到 Green
                         │
                         └──► Stream 2: 质量测试通过？
                              ├── 变异得分 ≥ 80%？
                              ├── APP Mass ≤ 阈值？
                              └── 规范合规 ≥ 80%？
                                   └── 否 → 回到 Refactor
```

## Enhancement 阶段

参考 Spec-First TDD，在 Ralph Loop 的 Refactor 之后增加可选 Enhancement 阶段：

```
Red → Green → CHECK → Refactor → [Enhancement] → 下一循环
                                     │
                                     ├── 检测测试弱点
                                     │    └── 伪变异扫描 → 补强断言
                                     │
                                     ├── 检测代码异味
                                     │    └── APP Mass 分析 → 建议
                                     │
                                     └── 跳过条件
                                          ├── 变异得分 ≥ 90%
                                          ├── APP Mass ≤ 3.0
                                          └── 时间 > 30min (防过度优化)
```

Enhancement 为可选阶段，满足跳过条件时自动跳过。
