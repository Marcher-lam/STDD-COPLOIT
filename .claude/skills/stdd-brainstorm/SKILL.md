# STDD Brainstorming 模式 (/stdd:brainstorm)

目标：提供"不改动任何代码"的纯分析和建议模式。AI 充当技术顾问，分析问题并提出方案，但**绝不修改文件**。

## 核心原则

> **"不要改变任何东西"** — 只分析、只建议、只讨论。

| 原则 | 说明 |
|------|------|
| 只读模式 | 不创建、不修改、不删除任何项目文件 |
| 多视角 | 从至少 3 个不同角度分析问题 |
| 方案对比 | 每个方案列出优劣和适用场景 |
| 可执行建议 | 输出可直接作为 `/stdd:propose` 输入的需求建议 |

## 执行步骤

### 1. 问题理解

从用户输入中提取：

- **核心问题**：用户想解决什么？
- **约束条件**：有哪些技术或业务限制？
- **上下文**：当前项目状态（从 `stdd/memory/` 和代码结构推断）

若信息不足，提问最多 **3 个**补充问题。

### 2. 多角度分析

从以下角度逐一分析：

| 角度 | 关注点 |
|------|--------|
| 架构视角 | 影响范围、模块边界、依赖关系 |
| 性能视角 | 时间复杂度、资源消耗、可扩展性 |
| 安全视角 | 数据安全、权限控制、攻击面 |
| 用户体验 | 交互流程、错误提示、可发现性 |
| 可维护性 | 代码复杂度、测试难度、文档需求 |

### 3. 方案生成

生成 **2-3 个**可行方案，每个方案包含：

```markdown
## 方案 A: <名称>

### 思路
<一句话描述>

### 优点
- <优点 1>
- <优点 2>

### 缺点
- <缺点 1>
- <缺点 2>

### 适用场景
<什么情况下选择此方案>

### 预估工作量
- 微任务数: N 个
- 预估复杂度: 低/中/高
```

### 4. 方案对比矩阵

```
┌──────────┬──────────┬──────────┬──────────┐
│ 维度     │ 方案 A   │ 方案 B   │ 方案 C   │
├──────────┼──────────┼──────────┼──────────┤
│ 复杂度   │ 低       │ 中       │ 高       │
│ 性能影响 │ 小       │ 中       │ 大       │
│ 风险     │ 低       │ 中       │ 低       │
│ 扩展性   │ 一般     │ 好       │ 优秀     │
│ 工作量   │ 1-2 天   │ 3-5 天   │ 1 周     │
└──────────┴──────────┴──────────┴──────────┘
```

### 5. 推荐建议

给出明确的推荐：

```markdown
## 推荐: 方案 <X>

理由: <为什么推荐此方案>

如果用户同意，可直接进入开发:
  /stdd:propose <方案 X 的需求描述>

如果需要深入讨论:
  继续在当前对话中讨论细节
```

## 输出

所有分析输出**仅显示在对话中**，不写入任何文件。若用户决定采纳方案，再通过 `/stdd:propose` 正式进入开发流程。

## 可选: 分析报告归档

若用户需要留存分析结果，可手动指定归档：

```
/stdd:brainstorm --save
```

此时写入 `stdd/brainstorm/<topic>/analysis.md`。

## 与其他 Skill 的关系

```
/stdd:brainstorm ──► (用户采纳) ──► /stdd:propose
                  ──► (继续讨论) ──► /stdd:brainstorm (新一轮)
                  ──► (用户放弃) ──► 结束
```

## 熔断机制

| 情况 | 处理 |
|------|------|
| 用户需求过于模糊 | 最多 3 轮提问后给出基于假设的分析 |
| 分析涉及多个独立问题 | 建议拆分为多个 Brainstorm 会话 |
| 用户开始要求写代码 | 提醒切换到 `/stdd:propose` |
